Anatomy of the URL (1)

  • A hyperlink (HTML link) points to a URL (Uniform Resource Locator). Possible parts of the URL:
  • protocol://username:password@subdomain.host:port/path/filename#anchor?parameters
  • Description:
    • protocol: way of data transport, e.g. http, https, mailto, tel... (full list here)
    • username:password: rarely used, allows immediate login on some pages
    • subdomain: first part of the domain, e.g. www
    • host: domain where site is hosted, e.g. gadgets.com
    • port: rather technical, is a number the server listens to (default 80)
    • path: where the resource is located, e.g. /products/jewelry/
    • filename: file name of the resource, e.g. overview.html
    • anchor: a link to a specific location on the page (see further)
    • parameters: data added to the URL, e.g. category=men&sort=price&order=asc

Anatomy of the URL (2)

  • Spot the parts in these examples:
  • http://www.gadgets.com/products/jewelry/list.php?category=men&sort=price
    https://login.gadgets.com:8080/?returnto=jewelry
  • For parts that aren't provided, default values are used:
  • https://gadgets.com/shops/ → https://www.gadgets.com:80/shops/index.php

Relative links

  • If no domain is provided, the link is relative; otherwise it is absolute
  • What is the URL, if the link comes from https://www.gadgets.com/sales/details/view.php?id=328?
  • /images/icon.png
    images/icon.png
    ../css/styles.css
  • Answers:
  • https://www.gadgets.com/images/icon.png
    https://www.gadgets.com/sales/details/images/icon.png
    https://www.gadgets.com/sales/css/styles.css
  • you should always use relative links for your own resources

Request / response (1)

  • Web pages are usually delivered through HTTP (HyperText Transfer Protocol)
    • user clicks on a link 'index.html'
    • browser sends HTTP request to the server for the page
    • server returns HTTP response with the html code
    • browser scans html code for any linked resource:
      images, css files, external scripts, fonts...
    • browser sends HTTP request for each resource
    • each received resource is inserted in the page

HTTP Request / response (2)

  • Learn by analyzing traffic with a network tool, e.g. for Toledo home (1, 2, 3, 4) and login (1)
  • Provides information about:
    • status codes (200 ok, 304 not modified, 404 not found...)
    • MIME types (text/html, text/css, image/png...)
    • timeline with load times and latencies
    • request and response headers
    • request methods GET and POST
    • cookies
    • ...

Load times

  • Observations:
    • first the html page is fully loaded
    • then other resources are loaded ± in order of appearance
    • number of parallel downloads is limited (usually 2 – 7)
  • Too many resources is bad for page loading time and your Google ranking
  • Possible solutions:
    • concatenate scripts & styles in large single files
    • use CSS sprite sheets for icons and other small images
    • ask your web hosting company to enable HTTP2

HTTPS

  • secure HTTP: all requests and responses are encrypted
  • server certificate ensures you are communicating with this site and not with a malicious copy
  • requires handshake / public&private keys / ... between server and browser (rather complex)
  • today an absolute must for websites
    • prevent stealing login credentials over plain HTTP with network sniffer
    • prevent MITM (Man In The Middle) attacks, injecting scripts etc.
    • plain HTTP sites are denied access to sensitive API's like webcam
    • HTTPS sites get a higher ranking in Google
    • performance loss is limited (a few %)
A WIFI network with a shared key where everyone knows the key isn't any safer than an open WIFI network. An attacker might create a copy of the network (same SID, same password) and then inject content in any HTTP website you visit. With HTTPS however, the certificate makes sure you are communicating with the website you want and not a copy, and on top of that, it also encrypts all your traffic.

Links in HTML

Examples

  • Example (notice the title attribute): 
    <a href="index.html"><h1>Site Title Here</h1></a>

<ul>
  <li>
    <a href="page1.html" title="edit this item">
      <img src="img/04_links/edit.gif" alt="edit icon">
    </a>
  </li>
  <li>
    <a href="page1.html" title="delete this item">
      <img src="img/04_links/del.gif" alt="delete icon">
    </a>
  </li>
</ul>
  • the (optional) title attribute adds meaning to the link
A good descriptive title attribute is especially important when linking an image, as screen readers (used by visually impaired) cannot read images.

Relative paths

  • Make all your internal links relative: 
    <!-- file 'index.html' -->
  ...
  <link rel="stylesheet" href="css/styles.css">
  <link rel="stylesheet" href="https://fonts.googleapis.com/css?family=Lato">
  <script src="js/lib/require/require.js"></script>
</head>
<body>
  ...
  <p><img src="img/pictures/photo.jpg" alt="my picture"></p>
  ...
    /* file 'css/styles.css' */
h1 {
  background-image: url(../img/logo.jpg);
  ...
}

Anchors

  • Are internal links that jump to another location in the same page
  • Link to id="someId" with href="#someId": 
    ...
<h2><a id="top">Jump to a specific chapter:</a></h2>
<ul>
  ...
  <li><a href="#chapter3">chapter 3</a></li>
</ul>
...
<h3><a id="chapter3">Chapter 3</a></h3>
<p>
  Lorem ipsum dolor sit amet, consectetur adipisicing elit...
  <a href="#top">back to top</a>
</p>
...

  <area shape="circle" coords="297,139,26" href="australia.html"
      alt="Australia">
   ...
</map>
  • Link to id="mapId" with usemap="#mapId": 
    <img src="img/adventureMap.gif" usemap="#globe" alt="world map">
    demo (Firefox)
  • most browsers dropped support; you shouldn't use image maps anymore
